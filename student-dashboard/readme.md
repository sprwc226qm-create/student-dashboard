## 一、项目概述

### 1.1 项目背景

随着高校数字化转型的加速推进，学生行为数据采集手段日益丰富——从教务系统的成绩数据，到一卡通的消费数据，再到图书馆的借阅记录和学习平台在线行为数据，海量数据已经沉淀下来。然而，"数据丰富、洞察贫乏"的矛盾依然突出：

- 辅导员无法快速了解班级学情全貌
- 学业预警依赖经验判断，缺乏数据驱动
- 学生个体画像缺失，个性化干预无从下手
- 数据散落在多个系统，缺乏统一的可视化入口

本项目以 Vue 3 前端技术栈为核心，构建了一个**学生行为数据可视化与学业分析平台**，将多源学生行为数据整合为直观、可交互的数据视图，帮助教学管理人员快速洞察学情、识别风险、精准干预。

### 1.2 项目目标

| 目标维度     | 具体描述                                                                                   |
| ------------ | ------------------------------------------------------------------------------------------ |
| **能力目标** | 掌握前端可视化技术（ECharts 6.x）、响应式布局、实时数据推送模拟、文件导出等核心前端能力      |
| **工程目标** | 独立完成一个中等复杂度前端项目的架构设计、组件开发与联调集成，共 40+ 源文件，13 条路由        |
| **思维目标** | 培养数据思维、产品思维、工程思维，理解"技术服务于业务"                                     |
| **素养目标** | 通过 REAL 能力模型，提升研究（Research）、工程（Engineering）、分析（Analysis）、领导（Leadership）四维能力 |

## 二、项目定位与设计思路

### 2.1 用户角色与核心场景

| 用户角色     | 核心场景                                                                           | 对应模块             |
| ------------ | ---------------------------------------------------------------------------------- | -------------------- |
| **学院领导** | 打开大屏，一眼看到全校学生行为概览——图书馆在馆人数、食堂消费高峰、各年级学习投入对比 | 模块一：数据大屏     |
| **辅导员**   | 查看某个学生的全面画像，了解其学习习惯、成绩趋势、消费行为，发现异常模式             | 模块二：学生画像     |
| **辅导员**   | 收到高风险预警推送，快速定位预警学生，查看详情并进行处理                             | 模块三：预警面板     |
| **辅导员**   | 为预警学生生成个性化干预报告，导出 PDF 发给家长或在会议上讨论                        | 模块四：干预报告     |
| **管理员**   | 通过自然语言查询系统数据，获取自动化分析结果与建议                                   | 模块五：AI 智能问答  |
| **移动办公** | 手机上收到告警，打开 H5 页面查看预警详情、学生画像，快速做出干预决策                  | 模块六：H5 适配      |

### 2.2 整体设计理念（数据驱动 + AI 增强）

本项目遵循两条核心设计原则：

1. **数据驱动（Data-Driven）**：所有页面组件通过 Pinia Store 获取数据，视图层仅负责渲染。Mock 数据层模拟真实后端 API 的响应格式，便于后续无缝对接真实接口。

2. **AI 增强（AI-Enhanced）**：在自然语言查询、个性化干预建议生成、异常趋势分析等场景中引入 AI 能力，以模拟 SSE 流式输出的方式提供智能问答体验。

系统整体遵循"一页一模块、模块内聚、跨模块共享"的组件化设计原则。所有可复用能力（图表封装、统计卡片、预警组件等）抽取为独立组件，避免重复开发。

## 三、总体方案设计

### 3.1 前端技术架构

系统采用分层架构设计，自上而下分为四个层次：

```
┌─────────────────────────────────────────────────────┐
│                    页面层 (Pages)                     │
│  dashboard/  portrait/  warning-center/  report/    │
│  ai-chat/   home/  practice/  course-charts/       │
├─────────────────────────────────────────────────────┤
│                   共享层 (Components)                │
│  charts/BaseChart  charts/HeatmapChart              │
│  dashboard/DataCard  dashboard/RealTimeTicker      │
│  warning/AlertNotification  common/WarningCard      │
├─────────────────────────────────────────────────────┤
│                    数据层 (Stores + Mocks)           │
│  stores/dashboard  stores/student  stores/warning   │
│  stores/report    stores/ai     stores/course       │
│  mocks/data/dashboard  mocks/data/warning          │
│  mocks/data/students   types/*.ts                  │
├─────────────────────────────────────────────────────┤
│                    路由层 (Router)                   │
│  13 条路由，懒加载，Web 端侧边栏导航                  │
└─────────────────────────────────────────────────────┘
```

**页面层**：每个业务模块对应一个或多个页面组件，负责组装子组件并编排交互逻辑。

**共享层**：跨模块复用的 UI 组件，按功能领域组织（charts/、dashboard/、warning/、common/）。

**数据层**：基于 Pinia 的状态管理，每个核心模块有独立的 Store。TypeScript 接口定义在 `types/` 目录下。Mock 数据模拟完整的后端响应，与 Store 解耦。

**路由层**：Vue Router 5.x，全部使用动态 `import()` 实现路由级代码分割，首页加载速度优化。

系统共设计 13 条路由，按模块组织：

| 路由路径          | 页面名称         | 所属模块     | 说明                     |

| `/portrait/:id`   | PortraitDetail   | 模块二       | 学生画像详情（动态路由）  |
| `/portrait`       | PortraitDefault  | 模块二       | 画像入口，重定向至 id=1   |
| `/home`           | Home             | 数据管理     | 学生成绩一览             |
| `/practice`       | Practice         | 数据管理     | 课程管理                 |
| `/course-charts`  | CourseCharts     | 数据管理     | 课程数据可视化           |
| `/chart-demo`     | ChartDemo        | 数据管理     | ECharts 图表演示         |
| `/warning-rules`  | WarningRules     | 模块三       | 预警规则配置             |

### 5.2 关键实现方案

#### 5.2.1 ECharts 图表统一封装

`BaseChart.vue` 是一个通用图表容器组件，封装了 ECharts 实例的创建、更新与销毁生命周期：

```ts
// 核心接口
defineProps<{
  option: EChartsOption   // 图表配置
  height?: string         // 高度，默认 100%
}>()
```

- 通过 `watch(option, deep: true)` 实现配置变更自动重绘
- `onBeforeUnmount` 中 `dispose()` 释放资源
- 全局 `resize` 事件监听实现自适应
- 所有页面通过传入 `computed` 的 option 来渲染图表，无需重复管理 ECharts 生命周期

#### 5.2.2 SSE 流式对话模拟

AI 对话采用 `setInterval` + 逐字追加的方式模拟 SSE 流式输出：

```ts
// stores/ai.ts - simulateStreamReply()
const streamTimer = setInterval(() => {
  if (index < fullText.length) {
    const chunk = fullText.slice(index, index + Math.floor(Math.random() * 4) + 2)
    index += chunk.length
    msg.content += chunk   // 逐字追加到响应式消息对象
  } else {
    clearInterval(streamTimer)
    isStreaming.value = false
  }
}, 30 + Math.random() * 50)  // 随机间隔模拟网络延迟
```

- `isStreaming` 状态控制发送按钮禁用、输入框 disabled、打字指示器显示
- `computed` 监听 `currentSession.messages` 实现响应式消息列表渲染

#### 5.2.3 Mock 数据层设计

所有数据通过 `src/mocks/data/` 目录管理，使用纯函数生成模拟数据：

- 支持参数化数据生成（如按年级筛选的学习趋势）
- 模拟随机波动（刷新时略微变化数值）
- 数据格式与预期的后端 API 响应保持一致，便于后续替换

#### 5.2.4 WebSocket 模拟（实时预警推送）

使用 `setInterval` 模拟 WebSocket 推送：

- 每 15 秒调用 `generateAlertNotification()` 生成新的预警通知
- 通知通过 `AlertNotification` 组件以 `el-popover` + `el-badge` 形式展示
- 每 2 秒生成一条实时行为事件，通过 `RealTimeTicker` 组件无缝滚动展示
- 组件 `onBeforeUnmount` 时清理定时器

### 5.3 移动端适配方案

## 八、问题反思与优化方向

### 8.1 已解决的问题

| 问题                         | 原因                                        | 解决方案                                                   |
| ---------------------------- | ------------------------------------------- | ---------------------------------------------------------- |
| AI 不回复消息                | `displayMessages` 使用普通变量而非响应式    | 改为 `computed(() => store.currentSession?.messages)`      |
| AI 对话 `sessionList` 不更新 | 同上，`sessionList` 非响应式                | 改为 `computed(() => store.sessions)`                      |
| jsPDF 构建报错               | 缺少 `canvg` 依赖                           | `npm install canvg`                                       |
| TypeScript 编译错误（12 个） | 未使用变量、缺失索引类型                    | 逐一修复：移除未使用声明、补全类型索引、使用 `_param` 前缀 |

### 8.2 已知限制

1. **Mock 数据静态性**：当前所有数据为前端生成的模拟数据，刷新后重置。后续可接入真实后端 API 或 MSW（Mock Service Worker）实现持久化。
2. **AI 意图匹配简单**：当前使用关键词匹配（6 种意图），覆盖有限。后续可接入真实 LLM API（如 Claude API）实现真正的自然语言理解。
3. **大屏分辨率**：当前主要优化 1920×1080 和 1366×768，超宽屏或 4K 屏幕可能存在留白。
4. **PDF 导出为截图**：当前 PDF 使用 html2canvas 截图方式，非矢量 PDF。后续可改进为使用 jsPDF 原生 API 逐元素绘制。
5. **国际化缺失**：当前仅支持中文，未引入 i18n 方案。

### 8.3 后续优化方向

| 方向                  | 描述                                                                                 | 优先级 |
| --------------------- | ------------------------------------------------------------------------------------ | ------ |
| **真实后端对接**      | 将 Mock 数据层替换为 RESTful API / GraphQL，接入真实数据库（PostgreSQL/MySQL）         | 高     |
| **LLM 集成**          | 接入 Claude API 或 GPT API，实现真正的自然语言数据查询与分析                           | 高     |
| **WebSocket 真实推送**| 替换 setInterval 模拟为真实的 WebSocket / SSE 连接，实现服务端推送                     | 中     |
| **PWA 支持**          | 添加 Service Worker + manifest.json，实现离线缓存与桌面安装                            | 中     |
| **单元测试**          | 使用 Vitest + Vue Test Utils 覆盖核心 Store 逻辑与关键组件                             | 中     |
| **E2E 测试**          | 使用 Playwright 覆盖关键用户流程（大屏浏览→画像查看→预警处理→报告导出）                | 低     |
| **数据权限控制**      | 实现角色权限管理（管理员/辅导员/领导），不同角色可见数据范围不同                         | 低     |
| **主题切换**          | 支持亮色/暗色主题自由切换（当前大屏固定暗色，其他页面固定亮色）                          | 低     |
